End-to-End Testing Guide

This guide covers the full Aigers Universe platform from login through projects, team members, workflow building, workflow execution controls, HITL, reporting, and AIger Copilot.

Landing and login Projects and team members Marketplace and agents Workflow Builder and run page Pause, resume, stop, and start Copilot grounding and trace

Before You Start

Confirm backend and frontend are running and required environment values are configured.

cd backend
uvicorn server:app --reload --host 0.0.0.0 --port 8001

cd frontend
yarn start

1. Entry and Sign-in

  1. Open the landing page at http://localhost:3000.
  2. Verify the updated app branding shows Aigers Universe and the hexagon icon.
  3. Click Enter workspace or Log in.
  4. Sign in using a valid user flow.
  5. Confirm you land on the dashboard.

Expected: no layout overlap, correct branding, and successful authentication.

2. Smoke Check Core Pages

  1. Open Dashboard.
  2. Open Projects.
  3. Open Marketplace.
  4. Open Agents.
  5. Open Workflow Builder.
  6. Open AIger Copilot.
  7. Open HITL.
  8. Open Observability.

Expected: each page loads without crash or blank-state blocker.

3. Projects and Team Members

  1. Open Projects.
  2. Create a project named Wave 1.
  3. Add at least two team member email addresses.
  4. Save the project.
  5. Reopen the same project.
  6. Confirm team members, owner, and project metadata persist.

Expected: project creation and member assignment persist correctly and remain editable.

4. Marketplace and Agents

Install these agents

  • Java to Spring Boot Architect
  • Java to Python Service Translator
  • Risk Analyzer
  • Compliance Checker
  • Recommendation Advisor

Verify on Agents page

  • framework badge
  • model badge
  • tag rendering
  • remote A2A state when enabled
  • delete and export actions

Expected: install works, cards render correctly, and no broken duplicate state appears.

5. Remote A2A Validation

  1. Open Agents and find Local agent cards.
  2. Copy one local card URL.
  3. Create a new agent with A2A enabled = true and A2A mode = Remote.
  4. Paste the copied URL.
  5. Click Test remote card.
  6. Save the agent.

Expected: validation succeeds before save and the resulting agent shows remote A2A state.

6. Workflow Builder Setup

  1. Open Workflow Builder.
  2. Select project Wave 1.
  3. Verify the reduced header height looks compact and stable.
  4. Name the workflow Wave 1 Modernization.

Expected: project scope persists and the builder header remains compact without layout regression.

7. Orchestrator Plan Test

Use this prompt:

Modernize this Java monolith into Spring Boot services, assess migration risk, and produce a phased remediation backlog.
  1. Click Auto-build workflow.
  2. In the planner modal test Edit, Replan, and Reject.
  3. Generate the plan again.
  4. Click Accept.

Expected: the planner behaves correctly, closes on accept, and nodes animate onto the canvas in order.

8. Workflow Inputs and Knowledge Base

Run-scoped workflow inputs

  1. Add a text goal.
  2. Upload one local file.
  3. Import repo https://github.com/mybatis/jpetstore-6.

Reusable KB

  1. Upload one architecture or policy document into KB.
  2. Optionally import a different repo into KB.

Expected: workflow inputs and KB remain separate in UI and in execution behavior.

9. Manual Builder Test

  1. Drag two installed agents onto the canvas.
  2. Connect them with an edge.
  3. Open node config for each.
  4. Change prompt, model, tools, input bindings, and A2A mode.
  5. Save the workflow.

Expected: node edits persist and the workflow saves successfully.

10. Run a Modernization Workflow

  1. Use an accepted orchestrator-built workflow or your manual flow.
  2. Add a text goal, one Java file, one repo import, and optionally one KB document.
  3. Click Run workflow.

Check on the run page

  • active node focus
  • status progression
  • Pause requests pause while the current agent finishes
  • Resume becomes available when the run is paused
  • Stop works for running and paused states
  • Start appears for stopped runs and restarts from the valid next step
  • A2A messages stream live
  • latest A2A entry auto-expands
  • final report opens from the same run page

Expected output: architecture findings, risk findings, remediation backlog, and readable report content.

11. HITL Approval Flow

Use this prompt in an appropriate workflow:

Review this vendor contract for key extraction, operational risk, compliance issues, and a final approval recommendation.
  1. Upload a contract-like document.
  2. Wait for the run to pause.
  3. Open HITL.
  4. Approve one run and reject another.
  5. Use the run-page HITL shortcut as well.

Expected: approval resumes execution, rejection marks failure, and return-to-run keeps operator context.

12. AIger Copilot Test

Start a new chat in AIger Copilot mode and use this prompt:

For a Java modernization use case, which agents should I install, which MCP tools should I enable, and what workflow should I build in this platform?
  1. Check the left history rail and central chat surface.
  2. Verify the composer is compact and smaller than before.
  3. Verify the composer shows mode, model, tool, and file controls only.
  4. Select Auto tool and then a specific tool and confirm both states are respected.
  5. Attach 2 or 3 files and ask a follow-up.
  6. Start another chat in Knowledgebase RAG mode and ask for a grounded summary of citations, KB retrieval, and graph visualization.
  7. Click one citation pill and confirm the modal opens the underlying source content.
  8. Ask an intentionally out-of-scope question in Knowledgebase RAG mode and confirm it politely refuses.
  9. Rename one chat from history.
  10. Delete one chat from history.
  11. Start another chat in General Reasoning mode.

Expected: session persistence works, grounded responses stay accurate, citations open real content, and workflow execution controls are not shown in chat.

13. Copilot Reasoning and Live Trace

  1. Ask a prompt that likely triggers tools or grounding.
  2. While the response is running, confirm there is no extra waiting placeholder text.
  3. Open the reasoning section and confirm logs are visible by default.
  4. Collapse the reasoning section, then reopen it while the same response is still active.
  5. Click Open live trace.

Expected: reasoning logs update from the real backend stream and the live trace stays in sync.

14. Report Validation

  1. Open the final report immediately after any workflow completes.
  2. Check heading styling, code blocks, callouts, and citations.
  3. Confirm the report opens even if materialization finished just moments before.

Expected: no broken report state and no report dead end for the user.

15. Shared Project Validation

  1. Create one workflow under Wave 1.
  2. Verify the workflow stays associated with the project.
  3. Log in as a second team member or validate through the shared project visibility path.
  4. Confirm the member can find the same project and workflow.
  5. Run the workflow once more from the shared context.

Expected: project-scoped workflows and runs remain visible to allowed team members.

16. Settings, Theme, and Provider Models

  1. Open Settings from sidebar and from user card.
  2. Switch provider between Platform Gateway, OpenRouter, Groq, and NVIDIA.
  3. Add a key for one provider and click save.
  4. Confirm model catalog updates and default model can be selected.
  5. Switch theme to Light.
  6. Open AIger Copilot and confirm background/surfaces change from dark palette to light palette.

Expected: settings persist per user, models are discovered dynamically, and light theme applies to Copilot and global shell.

17. Knowledge Graph 2D/3D

  1. Open Knowledge Graph page.
  2. Verify main/sub/document/chunk nodes render.
  3. Toggle between 3D and 2D modes.
  4. Click one node to highlight all adjacent nodes and edges.
  5. Drag nodes and click Save layout, then refresh page.

Expected: semantic and structural edges stay connected across the graph, highlight works, and saved layout reloads correctly.

18. Chunking Strategy Validation

  1. Upload one document with section-aware-large.
  2. Upload one document with sliding-window.
  3. Upload one document with table-first.
  4. Upload one document with semantic-topic.
  5. Check document records and confirm chunk_strategy is stored.

Expected: all four strategies ingest successfully and are persisted in document metadata.
