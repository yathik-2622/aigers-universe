End-to-End Testing Guide

This page covers quick but deep end-to-end testing of the full AIger's Universe platform from the landing page through workflows, A2A, HITL, reports, and Copilot.

Landing page to sign-in Marketplace and Agents Workflow Builder and Runs A2A streaming HITL approvals Copilot live reasoning

Before You Start

Confirm backend and frontend are both running and your required environment values are configured.

cd backend
uvicorn server:app --reload --host 0.0.0.0 --port 8001

cd frontend
yarn start

1. Entry and Sign-in

  1. Open the landing page at http://localhost:3000.
  2. Verify hero, navigation, and CTA cards render without overlap.
  3. Click the workspace entry button such as Enter workspace or Login.
  4. Sign in using a valid app user flow.
  5. Confirm you land on the dashboard after authentication.

Expected: no layout collisions, login succeeds, and the dashboard opens cleanly.

2. Smoke Check Core Pages

  1. Open Dashboard.
  2. Open Marketplace.
  3. Open Agents.
  4. Open Workflow Builder.
  5. Open AIger Copilot.
  6. Open HITL.
  7. Open Observability.

Expected: each page loads without crash, blank blockers, or overlapping UI.

3. Marketplace and Agents

Install these agents

  • Java to Spring Boot Architect
  • Java to Python Service Translator
  • Risk Analyzer
  • Compliance Checker
  • Recommendation Advisor

Verify on Agents page

  • framework badge
  • model badge
  • tag rendering
  • delete/view actions
  • A2A state when enabled

Expected: install works, cards render correctly, and no duplicate or broken agent state appears.

4. Remote A2A Validation

  1. Open Agents and find Local agent cards.
  2. Copy one local card URL.
  3. Create a new agent.
  4. Set A2A enabled = true.
  5. Set A2A mode = Remote.
  6. Paste the copied card URL.
  7. Click Test remote card.
  8. Save the agent.

Expected: validation succeeds, a summary appears before save, and the saved card shows remote A2A state.

5. Workflow Builder Setup

  1. Create a project named Wave 1.
  2. Open Workflow Builder.
  3. Select project Wave 1.
  4. Name the workflow Wave 1 Modernization.

Expected: project scope persists and builder state remains stable.

6. Orchestrator Plan Test

Use this prompt:

Modernize this Java monolith into Spring Boot services, assess migration risk, and produce a phased remediation backlog.
  1. Click Auto-build workflow.
  2. In the planner modal test Edit, Replan, and Reject.
  3. Generate the plan again.
  4. Click Accept.

Expected: the planner modal behaves correctly, closes on accept, and nodes animate onto the canvas in order.

7. Workflow Inputs and Knowledge Base

Run-scoped workflow inputs

  1. Add a text goal.
  2. Upload one local file.
  3. Import repo https://github.com/mybatis/jpetstore-6.

Reusable KB

  1. Upload one architecture or policy document into KB.
  2. Optionally import a different repo into KB.

Expected: workflow inputs and KB stay separate and show separate lists and state.

8. Manual Builder Test

  1. Drag two installed agents onto the canvas.
  2. Connect them with an edge.
  3. Open node config for each.
  4. Change prompt, model, tools, input bindings, and A2A mode.
  5. Save the workflow.

Expected: node edits persist and the workflow saves successfully.

9. Run a Modernization Workflow

  1. Use an accepted orchestrator-built workflow or your manual flow.
  2. Add a text goal, one Java file, one repo import, and optionally one KB document.
  3. Click Run workflow.

Check on the run page

  • active node focus
  • status progression
  • A2A messages streaming live
  • latest A2A entry auto-expands
  • final report opens from the same run page

Expected output: architecture findings, risk findings, remediation backlog, and readable report content.

10. HITL Approval Flow

  1. Run a compliance-heavy workflow, for example using this prompt:
Review this vendor contract for key extraction, operational risk, compliance issues, and a final approval recommendation.
  1. Upload a contract-like document.
  2. Wait for the run to pause.
  3. Open HITL.
  4. Approve one run and reject another.
  5. Use the run-page HITL shortcut as well.

Expected: paused runs appear correctly, approval resumes execution, rejection marks failure properly, and return-to-run keeps context.

11. AIger Copilot Test

Start a new chat in AIger Copilot mode and use this prompt:

For a Java modernization use case, which agents should I install, which MCP tools should I enable, and what workflow should I build in this platform?
  1. Check the left history rail and central chat surface.
  2. Verify the composer shows mode, model, tool, and file controls.
  3. Change model and preferred tool using the smaller upward-opening dropdowns.
  4. Attach 2 or 3 files and ask a follow-up.
  5. Rename one chat from history.
  6. Delete one chat from history.
  7. Start another chat in General Reasoning mode.

Expected: session persistence works, history actions do not overlap, and grounded platform responses stay accurate.

12. Copilot Reasoning and Live Trace

  1. Ask a prompt that likely triggers tools or grounding.
  2. While the response is running, confirm there is no extra waiting placeholder text.
  3. Open the reasoning section and confirm logs are visible by default.
  4. Collapse the reasoning section, then reopen it while the same response is still active.
  5. Click Open live trace.
  6. Keep the modal open while the backend continues processing.

Expected: reasoning logs update dynamically from the real backend stream, reopening shows the current logs, and the live trace modal keeps showing current internal activity.

Also verify: response tags for Mode, Model, and Tool appear only after the response completes.

13. Report Validation

  1. Open the final report immediately after any workflow completes.
  2. Check heading styling, code blocks, callouts, and citations.
  3. Confirm the report opens even if materialization finished just moments before.

Expected: no broken report state and no delay-based dead end for the user.

15. Repo + Design Document Flow

  1. Import one GitHub repo into Workflow inputs.
  2. Upload one design document or architecture note into Workflow inputs.
  3. Use this prompt:
Use this design document and the imported repository together to identify architecture drift, modernization risks, target boundaries, and rollout phases.

Expected: the repo and document remain separate inputs and the output reflects both sources.
